When structured logging is enabled via `Instrumentation.Log.Format`, ZITADEL writes structured log records to standard error.
This page describes the fields, formats, and conventions used in the log output.

## Enabling structured logs

Set the `Format` field to one of the supported formats to activate structured logging:

```yaml
Instrumentation:
  Log:
    Format: "json" # or "text", "gcp", "gcp_error_reporting"
```

When `Format` is `disabled` (the default), ZITADEL falls back to the legacy `Log` configuration.

## Log formats

| Format | Output | Best for |
|---|---|---|
| `text` | Human-readable key=value pairs on stderr | Local development, manual debugging |
| `json` | JSON objects, one per line | Machine parsing, log aggregation pipelines |
| `gcp` | JSON with Google Cloud Logging fields (`severity`, `message`, `logging.googleapis.com/sourceLocation`) | Google Cloud Logging integration |
| `gcp_error_reporting` | JSON with Google Cloud Error Reporting fields (`@type`, `serviceContext`) | Google Cloud Error Reporting integration |

## Common log fields

Every structured log record contains the following fields.
Field names follow the Go `slog` conventions.

| Field | Type | Description |
|---|---|---|
| `time` | string (RFC 3339) | Timestamp when the log record was created |
| `level` | string | Log severity: `DEBUG`, `INFO`, `WARN`, or `ERROR` |
| `msg` | string | Human-readable log message |
| `source` | object | Source file location where the log was emitted (when `AddSource: true`) |
| `source.function` | string | Go function name |
| `source.file` | string | Source file path |
| `source.line` | number | Line number in the source file |

### Request log fields

Log records in the `request` stream include additional fields for HTTP and gRPC requests:

| Field | Type | Description |
|---|---|---|
| `protocol` | string | `http` or `grpc` |
| `method` | string | HTTP method (e.g., `GET`, `POST`) or gRPC method path |
| `host` | string | Request host header |
| `path` | string | Request path |
| `status` | number | HTTP status code or gRPC status code |
| `duration` | number | Request processing time in milliseconds |
| `request_id` | string | Unique request identifier |
| `instance_id` | string | ZITADEL instance that handled the request |
| `auth_method` | string | Authentication method used, if applicable |

### Error fields

When an error is logged, additional fields provide diagnostic context:

| Field | Type | Description |
|---|---|---|
| `error` | string | Error message |
| `error.id` | string | ZITADEL error identifier (e.g., `USER-Gsj9q`) |
| `error.location` | object | Source location where the error was created (when `ReportLocation: true`) |
| `error.location.file` | string | Source file |
| `error.location.line` | number | Line number |
| `error.location.function` | string | Function name |
| `error.stack_trace` | string | Full stack trace (when `StackTrace: true`) |

### Event handler fields

Log records in the `event_handler` stream include fields about projection processing:

| Field | Type | Description |
|---|---|---|
| `handler` | string | Name of the event handler / projection |
| `event_type` | string | Type of event being processed |
| `aggregate_type` | string | Aggregate type of the event |
| `aggregate_id` | string | Aggregate ID of the event |
| `sequence` | number | Event sequence number |

## Streams

Streams control which subsystems emit log records.
Each stream produces logs for a specific area of the application:

| Stream | Description | Typical use |
|---|---|---|
| `runtime` | Startup, shutdown, and general lifecycle messages | Always useful; confirms ZITADEL started correctly |
| `ready` | Readiness and health check results | Health monitoring; used by the `zitadel ready` command |
| `request` | Incoming API and HTTP requests with response status | Access logging, traffic analysis, debugging request flows |
| `event_handler` | Event processing in projections | Debugging projection delays or failures |
| `queue` | Job queue processing | Monitoring background job execution |
| `event_pusher` | Event pushing to the database | Deep debugging only — **contains sensitive data** |

<Callout type="warn">
The `event_pusher` stream outputs data that may contain personally identifiable information (PII).
Only enable it in controlled debugging scenarios, never in production.
</Callout>

Configure streams via YAML or as a comma-separated environment variable:

```yaml
Instrumentation:
  Log:
    Streams:
      - runtime
      - request
```

```bash
ZITADEL_INSTRUMENTATION_LOG_STREAMS=runtime,request
```

## Masking sensitive data

The masking configuration replaces attribute values matching specified keys.
This prevents sensitive data from appearing in logs or exported telemetry:

```yaml
Instrumentation:
  Log:
    Mask:
      Keys:
        - "first_name"
        - "last_name"
        - "email"
      Value: "****"
```

**How key matching works:**

- Keys are unqualified and match at any nesting depth.
  For example, `"email"` matches both `email` and `user.email`.
- When the key matches a **group** (nested object), all values in the group are masked but the structure is preserved.
- When the key matches a **non-group value** that contains a nested object or array, the entire value is replaced with the mask string.

## Example log output

### JSON format

```json
{
  "time": "2025-01-15T10:30:45.123Z",
  "level": "INFO",
  "msg": "request completed",
  "source": {
    "function": "github.com/zitadel/zitadel/internal/api.(*Server).Handle",
    "file": "internal/api/server.go",
    "line": 142
  },
  "protocol": "http",
  "method": "POST",
  "path": "/oauth/v2/token",
  "status": 200,
  "duration": 12.5,
  "request_id": "abc123",
  "instance_id": "123456789"
}
```

### Text format

```
time=2025-01-15T10:30:45.123Z level=INFO msg="request completed" source.file=internal/api/server.go source.line=142 protocol=http method=POST path=/oauth/v2/token status=200 duration=12.5
```

## OTEL log export

In addition to printing logs to stderr, ZITADEL can export log records to an OpenTelemetry backend.
This is configured separately from the stderr format and can run in parallel:

```yaml
Instrumentation:
  Log:
    Format: "json"        # Print to stderr
    Exporter:
      Type: "grpc"        # Also export via OTEL
      Endpoint: "otelcol.monitoring.svc:4317"
```

See the [Instrumentation](/self-hosting/manage/instrumentation) page for full exporter configuration.
